6. BACKEND EXPLANATION
--------------------------------------------------------------------------------
How the server works:
The Express application initializes in `src/server.js`, binds to port 5001, and connects directly to the local/cloud MongoDB cluster using Mongoose.

Request-response flow:
When a client sends a request (e.g., fetching a product), the request hits the Express routing layer. If the route is protected, it processes through the `auth` middleware to validate the JWT. Upon successful validation, it queries MongoDB, formatting the results natively into JSON, and pipes them back to the client.

Static file handling (uploads/images):
The backend sets up an express.static directory linked to `../uploads`. When a vendor uploads a 3D model using `upload.js` (Multer), it catches both the primary model file (.gltf) and any accompanying binaries (.bin), creating an isolated directory under the products explicit ID structure, entirely shielding them from conflicting names.

9. CHALLENGES FACED
--------------------------------------------------------------------------------
Technical difficulty: 
Handling 3D assets on standard HTTP networks. Unlike single-image uploads, 3D GLTF models often rely on multiple companion files (.bin, .jpg textures) that must be preserved within the same folder space upon being passed through the multipart form data boundary.

How it was solved: 
Configured Multer to handle an array subset of files simultaneously, leveraging Node's local `fs` (Filesystem) API to dynamically create explicit directory folders tied uniquely to the `product._id`. The models are pushed synchronously into that dedicated path so the frontend viewer easily fetches the root `.gltf` and seamlessly resolves its children binaries inherently holding the model together.

11. ADVANTAGES OF THE PROJECT
--------------------------------------------------------------------------------
Why the structure is good:
The directory segregation (backend vs client, and `src/` modularization) represents industry-standard design. It prevents scope pollution, allows Frontend and Backend teams to work concurrently, and guarantees components and routes represent their exact functions.

Maintainability and scalability benefits:
Relying mechanically on decoupled components and MongoDB NoSQL allows the data architecture to pivot effortlessly—an excellent safety net if requirements suddenly shift mid-production without forcing heavy schema migrations.
